Methods are incredibly useful. They allow us to perform an action in the programming world. If objects are the sandwiches of the world, then methods are the tools with which we modify the sandwich from a mediocre PB&J to a multi-layer beast which Dagwood would be proud to eat.

When creating methods, there are two main types of methods to consider: class methods and instance methods. Simply, class methods act on the class itself; instance methods act upon an instance of the class. Both types of methods have pros and cons. First, though, let's understand the differences of the syntax and implementation between class and instance methods.

Creating and Implementing Class Methods

There are many different ways to create a class method. Here are the two most common:

class SayingHello

  def self.hello1
    puts "The class says hello from method 1."
  end

  class << self
    def hello2
      puts "The class says hello from method 2."
    end
  end
end

SayingHello.hello1
SayingHello.hello2

### OUTPUT ###
The class says hello from method 1.
The class says hello from method 2.

Pretty straightforward, right? The first method is a little easier to read for one method, but the second method is especially useful when defining multiple class methods within a class, making it DRY-friendly.

Creating and Implementing Instance Methods

An instance method is one of the most popular ways to create a method in Ruby. As such, it doesn't need much explanation. Regardless, here is an example so you can compare syntax.

class SayingHello
  def hello
    puts "An instance of the class says hello."
  end
end

example = SayingHello.new
example.hello

### OUTPUT ###
An instance of the class says hello.

Comparing the Syntax of Class and Instance Methods

It's pretty easy to see why the above methods work, but it's sometimes difficult to compare the syntax. Here's an example of Class and Instance methods side-by-side:

class SayingHello

  def self.hello      # class method
    puts "The class says hello."
  end

  def hello           # instance method
    puts "An instance of the class says hello."
  end

end

SayingHello.hello                     # using the class method
instance_hello = SayingHello.new      # creating an instance
instance_hello.hello                  # instance method

### OUTPUT ###
The class says hello.
An instance of the class says hello.

As you can see, the class method acts directly on the class. Conversely, an instance method only works on an instance of the class; thus, you must create an instance (SayingHello.new) to use these methods.

It's important to note that any information used in a class method must be passed through the method itself. Conversely, an instance of the class can accept information through both the initialization method and any instance method.

class SayingHello

  def initialize(person)
    @person = person
  end

  def self.hello(person)
    puts "#{person} says hello from the class."
  end

  def hello
    puts "#{@person} says hello from an instance of the class."
  end

  def hello_special(type = nil)
    case type
    when "english"
      puts "#{@person} says it is a pleasure to meet you."
    when "australian"
      puts "#{@person} says 'G'day, mate!"
    when "pirate"
      puts "#{@person} says 'Arrrrrrrr!'"
    else
      puts "#{@person} says hello."
    end
  end
end

SayingHello.hello("Cari")
cari = SayingHello.new("Cari")
cari.hello
cari.hello_special("pirate")

### OUTPUT ###
Cari says hello from the class.
Cari says hello from an instance of the class.
Cari says 'Arrrrrrrr!'

Pros and Cons of Class Methods

Class methods are pretty snazzy because they allow you to act immediately. It results in shorter code since there is no need to initialize an object to execute a method. This code can be more straightforward, easier to read, and extremely useful.

Class methods have many cons, though. If a complex method is necessary, it would be difficult to make a method single-purpose using a class method. Any parameters must be passed directly through the class method, which can be cumbersome as the number of parameters increases.

In addition, it is almost impossible* to keep track of an instance's state using a class method. This means that class methods require that all information necessary be passed through arguments in the class method. For simple class methods (especially any that require no arguments), this isn't a problem. However, this means that complex methods would be difficult to create because there is no easy way to relay all the information that needs to be passed to complete the action.

Using class methods emphasizes a procedural-like approach to programming. Since Ruby's strength is in object-oriented programming, this is inefficient and annoyingâ€” much like trying to cut your steak with a butter knife.

Pros and Cons of Instance Methods

The main con of an instance method is that it requires initialization of an object to be of any use. If the action is simple, then initialization can result in lengthy and indirect code.

On the other hand, the greatest strength of an instance method is that it tracks the state of an intialized object. This means that the object's state can and will influence future method calls. The user has to keep track of less information, and DRY & single-purpose code is easier to create.

Tracking an object's state is important in object-oriented program. It allows you to perform complex actions by calling multiple methods on the same object. If you need to call multiple arguments in a specific order, it means you can call those methods and arguments in a logical and intuitive way. The more complex a class, the more you must be able to track an object's state.

Concluding Thoughts

Both class methods and instance methods have their time and place in Ruby. However, class methods emphasize a procedural-like implementation of Ruby, which can increase maintenance necessary as codebase increases. They are useful when initialization of a class introduces an extra and unnecessary step, but they should be used carefully and sparingly.

When in doubt, it is probably better to use instance methods. Instance methods allow you to track the state of an object, which allows for more complex classes which are more easily maintained in the long run.

*Nothing is impossible. But things which are cumbersome and circuitious should be abandoned as useless, if not quite impossible.
