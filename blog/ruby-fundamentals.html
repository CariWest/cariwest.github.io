Once upon a time, I was writing a method, and I was having trouble with the variables. My friend said "Just use the .dup method," and my method worked - but I had no idea why, no matter how my friend tried to explain why it was necessary. After reading "The Well-Grounded Rubyist," though, a light bulb went off. I get it now.

The issue, I realize, is that I've been thinking about objects from the wrong side of the = symbol.

To delve into this problem, let's create a string object.

string_variable = "string object"
=> "string object"

In the example above, the object is the string. It is on the right hand side. The object is not the variable. I don't know how many different ways I heard or read this information when I was teaching myself Ruby, but somehow it never clicked. But look again - the object is on the right hand side.

Of course, the way we manipulate objects can be a little misleading in this regard. For example, when you want to know the object's id (a unique identifier which each object has), you ask the variable to get the information for you.

string_variable.object_id
=> 700980975920

A Tale of Two Variables - But Only One Object

Okay, so we know the object id. Why is that important? Well, what if you assigned a different variable the same string? You can assign a new variable to the same string using the variable you already have. This will cause your new variable to point to the same object. You can tell this is true when you look at the object id, which is the same.

other_variable = string_variable
=> "string object"
other_variable.object_id
=> 700980975920

See? Do you get it?! They're the same! Both string_variable and other_variable are pointing to the same object. This is a smart feature of Ruby's design, because it means that Ruby doesn't make multiple copies of the same object. That would be a waste of programming space.

Reassigning Variables

But wait, there's more! As long as a variable points to that object, it remains the same. If you reassign one of the variables, it will merely point to a different object.

string_variable = "Hello World"
=> "Hello World"
puts "string_variable: #{string_variable.object_id}"
=> string_variable: 70098098671640
puts "other_variable: #{other_variable.object_id}"
=> 700980975920

As you can see, other_variable still points to the same object (as evidenced by the object id), but string_variable now points to a different object.

Here's another way to look at it. Imagine that you and your friend are attending a weird psychology study, and you are assigned a task: you must point at the pysical object you are told to point to by an Operator - but you can only change what you're pointing to when the Operator addresses you directly. The Operator tells your friend to point at the table. The Operator tells you to point at the same thing your friend is pointing at, so you point at the table. Now the Operator tells your friend to point at the chair. The Operator hasn't addressed you, so you are still pointing at the table. In this scenario, you and your friend are variables, while the table and the chair are objects. I could go on and on with this analogy (I haven't even explained how methods worked!), but I think the gist of it is pretty clear.

Replacing an Object

So, say you want to replace the object with something else, but you want keep the same object id? You can do that with the replace method!

other_variable.replace("updated string object")
=> "updated string object"
other_variable.object_id
=> 700980975920

It's pretty sneaky, hmm? This can be useful if you want to change the object but you want to keep the object ID. Of course, if two variables are pointing to the same object, keep in mind that replacing the object will change the object for both variables.

There's so much more to learn...

Unfortunately, we've fulfilled the scope of this blog post. However, Objects are so much more nuanced than what I've covered. I highly recommend reading "The Well-Grounded Rubyist" by David Black for a more thorough and nuanced discussion of Objects.

*Please note: if all variables are reassigned and nothing is pointing at the object, Ruby will decide to "forget" it's there. If the object is not being used, Ruby has no need to store it in the memory. If you recreate the object, it will then have a new object id.
